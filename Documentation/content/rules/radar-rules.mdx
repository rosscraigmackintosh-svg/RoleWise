# Radar rules
Radar uses a defined ruleset so discovery and summary stay calm, factual, and useful.
## Purpose
Radar summarises changes in the job search using logged data. It should highlight signal, not create urgency.
## Data sources
Radar works from:
- applications log
- stats
- recruiter intelligence
- weekly review
- decision patterns
## Non-negotiables
Radar must be:
- calm
- factual
- short
- non-speculative

It must not rank companies, predict outcomes, or guess intent.
## Time windows
Radar defaults to rolling and weekly views. These windows should remain explicit so the user knows what period they are looking at.
## Defined terms
Radar should use stable definitions for terms such as:
- stale
- ghosted
- hot
- blocked
- salary disclosed

These definitions should not shift casually.
## Output requirements
Radar should always surface:
- activity counts
- pipeline health
- response intelligence
- role mix signals
- risk flags
- one small next action
## Action rule
Radar should suggest only one simple next action at a time.
## In one sentence
Radar rules keep discovery and summary grounded in clear definitions and calm interpretation.
