# Coding expectation rule
Coding expectations are one of the clearest fit filters in Rolewise.
## Core rule
If a role requires production-level coding from the designer, it is a hard no.
This includes expectations such as:
- shipping production front-end code
- owning production UI implementation
- writing live application code as a core part of the role
## Acceptable coding
Coding is acceptable when it is clearly limited to:
- prototyping
- exploratory work
- concept validation
- rapid interaction testing
## Why this matters
The distinction is not technical. It is role-shape driven.
A designer who prototypes is still operating in a design-led role. A designer responsible for production front-end delivery is being asked to perform a different job.
## Unknown expectations
If coding language is vague, Rolewise should surface it as a verification point rather than assume the best.
## In one sentence
The coding expectation rule protects against roles that quietly shift from design into production engineering.
